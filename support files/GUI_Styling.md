# Font-Sensitive GUI Scaling in AmigaOS 3.0

## Dynamic Layout with GadTools and Intuition

AmigaOS 3.0 applications were expected to adapt their GUI to the user's chosen system font[\[1\]](https://amigaos.exec.pl/amiga_user_interface_style_guide/basics_of_the_gui.html#:~:text=Users%20can%20specify%20which%20font,in%20the%20Workbench%27s%20Prefs%20drawer). Since Workbench 3.0 uses only monospaced system fonts (proportional fonts weren't supported in 3.0)[\[2\]](https://amigaos.exec.pl/amiga_user_interface_style_guide/basics_of_the_gui.html#:~:text=Keep%20in%20mind%20that%20the,is%20guaranteed%20to%20be%20monospaced), programs could assume a fixed character width, but still needed to scale gadget sizes and positions to accommodate different font sizes. Developers typically retrieve the current screen or system font (e.g. via IntuitionBase->DefaultFont or a locked public screen's DrawInfo->dri_Font) and use its metrics to calculate dimensions. Common practice was to **express all gadget coordinates and sizes in terms of the active font's dimensions** rather than absolute pixels[\[3\]](https://eab.abime.net/showthread.php?t=121068#:~:text=Board%20eab,Thus%2C%20you%20can%20say)[\[4\]](http://amigadev.elowar.com/read/ADCD_2.1/Libraries_Manual_guide/node03DE.html#:~:text=,the%20left%20and%20right). This ensures the interface scales consistently when the user changes the Workbench font or screen resolution.

## Measuring Text for Gadget Width

To size labels and other text elements, developers use the graphics library's text measuring functions. The fundamental one is TextLength(), which returns the pixel width of a string in a given RastPort/font. For Intuition's higher-level IntuiText structures (often used for gadget labels), the analogous function is IntuiTextLength() - it returns the pixel length of an IntuiText string[\[5\]](https://wiki.amigaos.net/wiki/Intuition_Text#:~:text=To%20determine%20the%20pixel%20length,function). For example, if you have a label "Backup Location:", you would obtain a RastPort for the window or screen (e.g. via GetScreenDrawInfo()), set its font to the system font, then call TextLength(rp, "Backup Location:", 16) (16 characters) to get its width in pixels. Intuition will use the screen's font metrics for this calculation; if no font is specified, it falls back to the system DefaultFont (monospaced) by default[\[6\]](https://wiki.amigaos.net/wiki/Intuition_Text#:~:text=LONG%20IntuiTextLength%28%20struct%20IntuiText%20,).

In practice, **source code often loops through all gadget labels to find the longest and determine a uniform width**. For instance, an AmigaOS calculator tool iterates over an array of button labels and uses TextLength() to compute each label's width, tracking the maximum length to decide the button gadget width[\[7\]](https://en.wikibooks.org/wiki/Aros/Developer/Docs/Libraries/GadTools#:~:text=InitRastPort%28%26temprp%29%3B%20SetFont%28%26temprp%2Cdri). A snippet from that code illustrates this approach:

InitRastPort(&rp);  
SetFont(&rp, drawInfo->dri_Font); // use screen's font  
buttonHeight = drawInfo->dri_Font->tf_YSize + EXTRA_H;  
for(i=0; i<NUM_BUTTONS; i++) {  
int len = TextLength(&rp, labels\[i\], strlen(labels\[i\]));  
if(len > buttonWidth) buttonWidth = len;  
}  
buttonWidth += EXTRA_W; // add padding around text

Here tf_YSize is the font height, and EXTRA_H/EXTRA_W are small padding constants. The code ensures every button is wide enough for its text plus some margin[\[7\]](https://en.wikibooks.org/wiki/Aros/Developer/Docs/Libraries/GadTools#:~:text=InitRastPort%28%26temprp%29%3B%20SetFont%28%26temprp%2Cdri). (In this case, they added 8 pixels horizontally and 4 pixels vertically[\[8\]](https://en.wikibooks.org/wiki/Aros/Developer/Docs/Libraries/GadTools#:~:text=,4) as padding - roughly half a character width as a margin.)

## Font Metrics for Gadget Sizing

Amiga fonts provide metrics to guide gadget dimensions. Key fields of the TextFont structure include **tf_XSize** (nominal character width for fixed-width fonts), **tf_YSize** (font height in pixels), **tf_Baseline** (baseline offset from the top of a character), and **tf_BoldSmear** (additional width for bold styling)[\[9\]](https://en.wikibooks.org/wiki/Aros/Developer/Docs/Libraries/Graphics#:~:text=UWORD%20%20%20tf_XSize%3B%20,affect%20a%20bold%20enhancement). Using these metrics:

- **Widths:** For monospaced fonts, tf_XSize gives the width of each character cell[\[10\]](http://amigadev.elowar.com/read/ADCD_2.1/Libraries_Manual_guide/node03DE.html#:~:text=,the%20left%20and%20right). You can multiply this by a number of characters to compute gadget widths or spacing. For example, spacing between a label and its input field might be set to 2 \* tf_XSize (two characters' width) to ensure a consistent gap regardless of font size. In practice, many programs used small fixed offsets (e.g. 4-8 pixels) which happened to scale reasonably with common fonts. The Workbench style guide recommended designing for the _worst-case_ width (monospaced) and adjusting if you can handle proportional fonts[\[2\]](https://amigaos.exec.pl/amiga_user_interface_style_guide/basics_of_the_gui.html#:~:text=Keep%20in%20mind%20that%20the,is%20guaranteed%20to%20be%20monospaced).
- **Heights:** Gadget height is typically tied to the font's Y-size (overall pixel height). A common formula is gadget height = font.tf_YSize + K where _K_ is a constant padding for the gadget's frame. For standard GadTools string gadgets, buttons, and text fields, developers often add a few pixels above and below the text. For example, Commodore's preference programs often set gadget height to one line of text plus a couple of pixels. In the calculator example, each button's height was tf_YSize + 4[\[11\]](https://en.wikibooks.org/wiki/Aros/Developer/Docs/Libraries/GadTools#:~:text=,4), giving a bit of vertical padding. Similarly, a list view showing _N_ lines could be set to N \* (tf_YSize + lineSpacing) tall. The GadTools utility library _GadUtil_ documented that a listview of ~4 lines should calculate its height as 4 \* (font_height + spacing)[\[12\]](https://aminet.net/package/dev/gui/gutil_dev#:~:text=The%20LAYOUTA_Spacing%20tag%20for%20LISTVIEW_KIND,used%20for%20the%20GU_AddHeiLines%20tag), where spacing is an extra pixel or two between lines. This ensures list browser gadgets scale with the font.
- **Baselines:** The tf_Baseline value specifies how far down the baseline is from the top of text[\[9\]](https://en.wikibooks.org/wiki/Aros/Developer/Docs/Libraries/Graphics#:~:text=UWORD%20%20%20tf_XSize%3B%20,affect%20a%20bold%20enhancement). When aligning text in different gadgets (e.g. a static label vs. text inside a string gadget), matching baselines is crucial so the text appears on the same horizontal line. In Workbench 3.0, all GUI text typically uses the same font, so baseline alignment mostly involves accounting for gadget framing. For instance, a string gadget usually has a recessed frame; the text inside might start a couple of pixels from the gadget's top edge. If you place a label (perhaps as a TEXT_KIND gadget or IntuiText) to the left of the string gadget, you might position that label slightly lower (by 1-2 pixels) so that its text baseline aligns with the text in the string gadget. In code, this could mean:

labelTop = stringGadgetTop + 1; // adjust for the string gadget's inner offset

The exact offset depends on the Intuition defaults - often one pixel was enough for GadTools gadgets. By using the same tf_Baseline for both (since the font is the same) and adjusting for any border, you achieve a consistent alignment. In other words, **ensure the label's TopEdge plus its font baseline equals the string gadget's TopEdge plus its text's baseline position**. Developers derived these offsets either from Intuition constants or by simple testing.

## Gadget Placement and Spacing Conventions

Beyond raw sizes, AmigaOS 3.0 GUI code followed informal layout rules to make interfaces look consistent:

- **Standard Margins:** Windows typically include an inner border gap. Intuition's Screen and DrawInfo provide the default border sizes. For example, on a Workbench screen, screen->WBorLeft and WBorTop give the pixel offsets for the window's drawable area. A program might use winLeft = screen->WBorLeft and winTop = screen->WBorTop + font->tf_YSize + 1 to position the first gadget below the window title bar[\[13\]](https://en.wikibooks.org/wiki/Aros/Developer/Docs/Libraries/GadTools#:~:text=win_borderleft%20%3D%20scr). This formula accounts for the screen's menu-bar height (which is one font-high, plus a 1-pixel spacer). Using these values ensures gadgets start at the right position inside the window's borders.
- **Relative Coordinates:** Subsequent gadgets are often placed relative to previous ones. For example, if you have a label and a text field pair, you might do:
- label.ng_LeftEdge = windowBorder + INNER_MARGIN;
- field.ng_LeftEdge = label.ng_LeftEdge + labelWidth + H_SPACING;

where H_SPACING could be a small multiple of tf_XSize. Many developers chose a spacing of 1 or 2 characters width. In our example code, button gadgets were laid out in a grid using col \* (buttonWidth + BUTTON_SPACING_X) for the X offset[\[14\]](https://en.wikibooks.org/wiki/Aros/Developer/Docs/Libraries/GadTools#:~:text=ng.ng_GadgetID%20%3D%20bi) - with BUTTON_SPACING_X defined as 4 pixels[\[15\]](https://en.wikibooks.org/wiki/Aros/Developer/Docs/Libraries/GadTools#:~:text=,4). This uniform spacing was effectively half the default character width (8) and kept the GUI from looking too cramped or too sparse.

- **Consistent Padding:** Around interactive gadgets (string gadgets, listviews, buttons), a little padding is added so text doesn't touch the gadget border. The GadTools system often has a default of 1 pixel inset. In custom layouts, developers explicitly add a few pixels. For instance, after measuring the longest button text, the code added +8 pixels to buttonWidth[\[16\]](https://en.wikibooks.org/wiki/Aros/Developer/Docs/Libraries/GadTools#:~:text=buttonheight%20%3D%20dri) as padding (so each side of the text had ~4px free) and +4 to buttonHeight[\[17\]](https://en.wikibooks.org/wiki/Aros/Developer/Docs/Libraries/GadTools#:~:text=InitRastPort%28%26temprp%29%3B%20SetFont%28%26temprp%2Cdri). These values were chosen empirically but correspond to comfortable padding with an 8×(tf_YSize) font. Essentially, **padding values were often proportional to the font size** - e.g. 4 pixels for an 8-pixel high font is a 50% padding; a larger font would naturally demand slightly more padding in absolute terms.
- **Listview and Multi-line Spacing:** For vertical spacing in list gadgets, the font height is the base. By default a GadTools ListView will size each entry to the font's Y-size (so text lines just fit). Developers could specify extra line spacing via LAYOUTA_Spacing or related tags. For example, setting a listview spacing of 1 will make each line's slot tf_YSize + 1 pixels tall[\[18\]](http://www.theflatnet.de/pub/cbm/amiga/AmigaDevDocs/gadtools.html#:~:text=GTMX_Spacing%20%28UWORD%29%20,One%20of%20PLACETEXT_LEFT%2C%20PLACETEXT_RIGHT)[\[19\]](http://www.theflatnet.de/pub/cbm/amiga/AmigaDevDocs/gadtools.html#:~:text=LAYOUTA_Spacing%20,point%20font%20users%29.%20%28V36). If a program wanted a list gadget to show, say, 5 visible lines, it would set the gadget's height to 5 \* (tf_YSize + spacing). The GadUtil library explicitly used this formula: _Height = number_of_lines \* (font_height + spacing)_[\[12\]](https://aminet.net/package/dev/gui/gutil_dev#:~:text=The%20LAYOUTA_Spacing%20tag%20for%20LISTVIEW_KIND,used%20for%20the%20GU_AddHeiLines%20tag). This keeps list views from cutting off text descenders and gives a bit of breathing room between lines.

## Examples from Workbench 3.0 Utilities

Many classic Amiga utilities from the 3.0 era demonstrate these techniques:

- **Preferences Editors (IControl, etc.)** - These were GadTools-based GUIs that read the user's preferred screen font and adjusted gadget sizes accordingly. For instance, the Font preferences editor itself had to display font names in a list gadget, adapting to various font heights.
- **IconEdit** - The icon editor on Workbench 3.0 uses string gadgets for entering icon tooltypes. It measures the labels (like "Default Tool:") to align the text fields. The layout ensures the colons line up and the input boxes are wide enough for typical paths.
- **DiskMaster 2** - A third-party file manager that ran on 2.x/3.0, it used a fixed-width font for its file list. While not all such programs allowed font changes, those that did (or even just allowed different screen modes) would calculate the column widths using TextLength() so that the columns scaled with the font.
- **AmigaDOS "ConfigEd"** - This was a configuration editor tool that likely opened on the Workbench screen and thus had to respect the Workbench font. Although source isn't readily available, it would have used IntuitionBase->DefaultFont metrics to size its gadgets, since Commodore's style guide mandated font-sensitive design[\[1\]](https://amigaos.exec.pl/amiga_user_interface_style_guide/basics_of_the_gui.html#:~:text=Users%20can%20specify%20which%20font,in%20the%20Workbench%27s%20Prefs%20drawer).
- **Example - Calculator (AROS port)** - The AROS project reimplemented a calculator using classic GadTools for compatibility. Its code (compatible with OS3) shows how to do dynamic GUI: it locks the Workbench screen, gets the default DrawInfo font, and computes all gadget sizes from that. As shown earlier, it sets gadget widths based on TextLength of labels, gadget heights as font.tf_YSize + constant, and window dimensions as sums of those gadget sizes plus spacing[\[20\]](https://en.wikibooks.org/wiki/Aros/Developer/Docs/Libraries/GadTools#:~:text=buttonwidth%20%2B%3D%20BUTTON_EXTRA_WIDTH%3B)[\[21\]](https://en.wikibooks.org/wiki/Aros/Developer/Docs/Libraries/GadTools#:~:text=inner_winwidth%20%3D%20buttonwidth%20,2). This mirrors how an OS3.0 programmer would manually layout a GUI.

## Limitations on Workbench 3.0

While these methods enabled basic font-sensitivity, Workbench 3.0 had some notable limitations:

- **No Proportional Fonts:** As noted, the system **default** font in 3.0 was guaranteed monospaced[\[2\]](https://amigaos.exec.pl/amiga_user_interface_style_guide/basics_of_the_gui.html#:~:text=Keep%20in%20mind%20that%20the,is%20guaranteed%20to%20be%20monospaced). Users could not fully switch the interface to a proportional font until later OS versions (3.1 allowed it to some extent). This meant developers often assumed fixed character widths in calculations. Programs that tried to be future-proof could attempt to handle proportional text by measuring every string, but many 3.0-era tools simply relied on the fixed-width constraint.
- **Manual Layout Only:** There was no automatic layout engine in OS3.0. Gadgets had to be positioned with explicit coordinates. If the user picked an unusually large font, a window might not automatically re-flow the gadgets - it was on the programmer to ensure the window was made large enough. Common tags like WA_AutoAdjust (to let Intuition enlarge the window if gadgets don't fit) could help but were limited. Developers had to set minimum window sizes based on the font. For example, one might add a safety margin or clamp the font size to avoid an excessively large UI.
- **Screen Mode Dependencies:** If a program opened on a custom screen with a different resolution or font (some apps did this to ensure a particular look), the scaling logic had to be applied there as well. Most apps simply opened on Workbench to inherit its font and scaling. On high-resolution screens (e.g. interlaced or Productivity modes), the same font might appear smaller relative to screen pixels, but since calculations were in pixels, the layout scaled correctly in absolute terms (just visually smaller). There was no concept of DPI scaling in 3.0 - only font-size scaling.
- **Bold and Italic Text:** If the interface used custom text styles (bold labels, etc.), the tf_BoldSmear and similar properties needed consideration. A bold style font could be a pixel or two wider. Typically, however, GUI text in 3.0 remained in the standard style (no bold/italic) for labels to avoid complicating measurements.
- **Localization Concerns:** Workbench 3.0 introduced locale support, meaning labels could change length in different languages. Developers were encouraged to allow extra space or dynamically size gadgets for longer text. As one expert noted, _"limiting the string length is the bread and butter of localization"_ - Commodore's localization tools (catcomp) let developers specify a maximum length for each message to keep UIs from breaking[\[22\]](https://eab.abime.net/showthread.php?t=121068#:~:text=Board%20eab,the%20CBM). In practice, well-designed 3.0 programs measured each label (as described above) and sized gadgets to the _actual_ text length at runtime, making them naturally handle different languages as long as the window was wide enough for the longest translation.

In summary, **Workbench 3.0 GUIs achieved scalability by calculating every element's size and position based on the system font metrics**. Using TextLength()/IntuiTextLength() for widths, tf_YSize (plus padding) for heights, and spacing offsets derived from the font ensured that dialog elements like labels, string gadgets, and list views remained aligned and usable under various user preferences[\[7\]](https://en.wikibooks.org/wiki/Aros/Developer/Docs/Libraries/GadTools#:~:text=InitRastPort%28%26temprp%29%3B%20SetFont%28%26temprp%2Cdri)[\[12\]](https://aminet.net/package/dev/gui/gutil_dev#:~:text=The%20LAYOUTA_Spacing%20tag%20for%20LISTVIEW_KIND,used%20for%20the%20GU_AddHeiLines%20tag). While rudimentary by modern standards, these techniques - visible in many classic utilities' source code - allowed AmigaOS 3.0 applications to respect the user's font settings and screen mode for a consistent look and feel.
